## 🎯 The 7 Immutable Rules

**These constraints prevent design drift:**

1. ❌ **No Hardcoded Colors** → Use tokens (--color-critical, --color-success)
2. ❌ **No Arbitrary Font Sizes** → Use scale (text-xs, text-sm, text-base... text-4xl)
3. ❌ **No Arbitrary Spacing** → Use 4px multiples (4, 8, 12, 16, 20, 24, 32, 40px)
4. ❌ **No Font Weights Beyond 700** → Only 400, 500, 600, 700
5. ❌ **No Transitions Beyond 400ms** → Only 150, 200, 300, 400ms
6. ❌ **No ALL CAPS Except Status** → Sentence case default
7. ❌ **No Raw HTML** → Use primitives (Box, Text, Button, Card)

**Enforcement:** ESLint rules + Code review checklist + CI/CD validation

## 📊 The 5 Laws of 2026 Enterprise Dashboards

**Applied to every page:**

1. **Hierarchy > Density** - One headline per section, secondary metrics grouped
2. **Scanning Patterns** - Most critical info top-left, actions top-right
3. **Color Means Something** - Red=critical, Amber=warning, Green=success
4. **Typography as Primary Tool** - Use size/weight, not color, for hierarchy
5. **Progressive Disclosure** - Show what matters, hide what doesn't

## ❓ Frequently Asked Questions

**Q: Can I break the rules for special cases?**
A: No. If you find a case that requires breaking a rule, update the rule instead (after architect review).

**Q: What if the business wants a red success indicator?**
A: No. Red means critical/error. Violating semantic meaning breaks user trust. Find a different approach.

**Q: Can I use a custom font size that "just looks better"?**
A: No. All decisions go through tokens. If the scale doesn't work, update the scale (with approval).

**Q: How often do tokens change?**
A: Rarely. Tokens are updated when business requirements change, not on a whim. Monthly review cycle.

**Q: What if a token doesn't exist for what I need?**
A: Create it (in styles/tokens-2026.css), submit for architecture review, add to Figma, then use it.
